# UI Parity Roadmap

This document is the single source of truth for aligning mobile UI with the web experience. It defines what must match, what can diverge, and target parity levels per screen.

Assumes the web repo is available at `../quran-app` (used by `npm run sync:web-core`).

## Parity Targets

- **95–100%**: Must match layout, content order, and interactions; only platform-specific affordances allowed.
- **90–94%**: Minor visual or interaction differences allowed if they improve native usability.
- **80–89%**: Feature set must match, but layout can be adapted to native patterns.
- **Below 80%**: Requires explicit approval and documentation of why it diverges.

## Notes

- When adding new screens or features, update this table first.
- Any intentional divergence must be documented in the Notes column with rationale.
- Mushaf offline roadmap Chunks 0-10 are in place: mobile settings now track pack-based mushaf IDs and a discrete `mushafScaleStep`, mushaf packs participate in the download index with a SQLite install registry for active-version lookup, the bundled `unicode-uthmani-v1` base pack is shipped locally and bootstrapped as an installed offline pack on app startup, hosted mushaf packs have catalog fetch + versioned file install/delete/activate infrastructure under `src/core/infrastructure/mushaf/`, the page route resolves the active local pack version and loads page lookup/lines/verse-word metadata from bundled or installed payloads with no network dependency, bundled Unicode mushaf pages render natively offline with stepped text sizing, `webview` packs use the web exact preset tables plus fit-aware centered RTL reflow inside the native page shell, `qcf-madani-v1` installs into the same versioned local pack contract with per-page local fonts for offline exact rendering, and mushaf word taps now bridge back into the native verse-action/bookmark/planner/share/audio flows without breaking exact-mode text selection and copy.
